# 컨텍스트 로더

이 저장소에서 Codex 작업을 시작할 때 항상 이 스킬을 사용한다. 목표는 `docs/` 전체를 읽는 것이 아니라, 작업에 필요한 문서만 선택적으로 로딩하는 것이다.

Codex의 기본 책임은 계획 검증, 설계 리뷰, 디버깅 분석, 구현 후 1차 검증이다. 메인 구현과 최종 판단은 Claude Code가 맡는다. 코드 수정은 구현 후 1차 검증 단계에서 명백한 버그, 타입 오류, 누락 guard, 테스트 실패 원인의 국소 수정에 한해 제한적으로 수행한다.

## 항상 로딩

먼저 다음 파일을 읽는다:

1. `AGENTS.md` — Codex 부트스트랩과 저장소 규칙
2. `CLAUDE.md` — 오케스트레이터 워크플로우와 프로젝트 지도
3. `docs/README.md` — 문서 인덱스
4. `docs/ARCHITECTURE.md` — 라우트 그룹, 레이어 방향, 외부 의존성
5. `docs/tech-debt-tracker.md` — 알려진 부채, baseline, 마이그레이션 메모

## 작업 유형별 라우팅

항상 로딩 세트 이후에는 관련 있는 문서만 추가로 읽는다.

| 작업 신호 | 추가 로딩 |
| --- | --- |
| Supabase, auth, cache, `revalidateTag`, Server Action, DB query | `.claude/skills/supabase/SKILL.md` |
| SCSS, layout, className, responsive UI, design token | `.claude/skills/styles/SKILL.md` |
| 새 파일, 파일 이동, 컴포넌트 위치, barrel export | `.claude/skills/file-structure/SKILL.md` |
| 다단계 구현 계획 리뷰 | `docs/exec-plans/active/`의 관련 계획, 없으면 Claude Code에 EXEC_PLAN 작성을 요청 |
| 아키텍처 또는 라이브러리 결정 | `docs/decisions/README.md`와 관련 ADR |
| 기존 버그 또는 회귀 | 관련 active/completed exec plan, `docs/tech-debt-tracker.md`, 영향받는 소스 파일 |
| 외부 라이브러리 동작 | `docs/references/` 먼저 확인. 없거나 오래됐으면 공식 문서만 조회 |
| 현재 작업에서 수집한 외부 자료 | `docs/research/`의 관련 노트 |
| 구현 후 1차 검증 | active EXEC_PLAN, 변경 파일, `docs/tech-debt-tracker.md`, 검증 출력 |

## 로딩 예산

컨텍스트를 작게 유지한다:

- 세부 문서보다 인덱스를 먼저 읽는다.
- 현재 결정에 영향을 줄 수 있는 파일만 로딩한다.
- 긴 문서는 프롬프트에 그대로 붙이지 말고 요약한다.
- 문서가 오래됐거나 코드와 충돌하면 동작 기준은 코드를 신뢰하고, 작업 범위에 맞으면 관련 문서를 갱신한다.

## 작업 시작 체크리스트

파일을 수정하기 전에 다음을 확인한다:

- 작업 유형을 명시한다.
- 로딩한 컨텍스트 파일을 나열한다.
- 영향받을 가능성이 있는 레이어를 식별한다.
- EXEC_PLAN이 필요한지 판단한다.
- 구현 요청인지 리뷰/분석 요청인지 구분한다.
- 구현 후 1차 검증인지 구분한다. 이 단계라면 국소 수정 가능 범위와 반환해야 할 항목을 먼저 확인한다.
- 구현 요청 또는 1차 검증 요청이 명시되지 않았다면 코드 수정 대신 분석 결과와 권장 다음 단계를 반환한다.

## 협업 루프별 출력

### 계획 검증

다단계 작업의 계획 검증에서는 결론을 내기 전에 다음 5체크를 수행한다.

1. 가정(Assumptions)이 명시되어 있는가?
2. 비목표(Non-goals)가 명시되어 있는가?
3. 변경 범위가 요청과 직접 연결되는가? (창발적 추가가 없는가)
4. 성공 기준(Success Criteria)과 검증 명령이 구체적인가?
5. 새 추상화·새 라이브러리·데이터 흐름 변경이 과한가? (없어야 정상)

이후 반드시 다음 중 하나로 결론을 낸다:

- `PASS`: 계획대로 구현 가능
- `CHANGE_REQUEST`: 구현 전 계획 수정 필요 (5체크 중 어느 하나라도 미충족이면 기본적으로 CHANGE_REQUEST)
- `BLOCK`: 중요한 정보나 결정이 없어 구현 중단 필요

응답에는 계획 변경 사항, 숨은 리스크, 검증 공백을 분리해서 적는다.

### 구현 후 1차 검증

구현 diff를 검증할 때는 다음 순서로 처리한다:

1. 레이어 경계, 데이터 흐름, 스타일 토큰, Supabase 클라이언트 사용을 확인한다.
2. **외과적 변경(surgical changes)** 을 확인한다 — 변경된 각 파일이 현재 task와 직접 관련 있는가? 인접 코드 정리·포맷·이름 변경이 섞여 있는가? 이번 변경으로 생긴 unused import/변수만 제거되었고 기존 dead code는 보존되었는가? 인접 정리가 섞여 있으면 별도 작업 분리를 요청한다.
3. 명백한 버그, 타입 오류, 누락 guard, 테스트 실패 원인의 국소 수정만 직접 수행한다.
4. 계획 변경, 새 라이브러리, 데이터 흐름 변경, 인증/캐시/배포 정책 변경, 외과적 변경 위반은 직접 수정하지 않고 Claude Code에 반환한다.
5. 수정했다면 변경 파일, 수정 이유, 검증 결과, 남은 리스크를 마지막에 적는다.

## 핸드오프 출력 형식

Claude Code 또는 다른 오케스트레이터에게 컨텍스트를 반환할 때는 다음 형식을 사용한다:

```text
작업 유형:
로딩한 컨텍스트:
관련 규칙:
예상 파일/레이어:
알려진 부채 또는 리스크:
권장 다음 단계:
```

## 중단 조건

다음 상황에서는 진행을 멈추고 오케스트레이터에게 확인한다:

- 작업에 비밀값, 프로덕션 데이터, 파괴적 DB 변경이 필요하다.
- 필요한 컨텍스트가 없거나 서로 모순된다.
- 요청된 변경이 `docs/ARCHITECTURE.md` 또는 accepted ADR과 충돌한다.
- 검증 실패를 신규 회귀와 기존 부채 중 어느 쪽으로도 분류할 수 없다.
- 1차 검증의 국소 수정 범위를 넘어서는 코드 수정을 요구하는 흐름이 감지된다.
