# Gemini Code Assist 코드 리뷰 스타일 가이드

> **목적**: Gemini Code Assist가 PR 리뷰 및 코드 제안 시 참조하는 조직 표준 지침입니다.
> **피드백 언어**: 모든 코드 리뷰 코멘트와 제안은 **한국어**로 작성하세요.

---

## 1. 단일 책임 원칙 (Single Responsibility Principle)

### 규칙

- 함수는 **하나의 비즈니스 로직 단위**만 처리해야 합니다.
- 데이터 변환 로직과 입출력(I/O) 로직은 반드시 **별도의 비공개 메서드**로 분리해야 합니다.
- 하나의 함수 내에서 데이터 파싱, 유효성 검사, DB 저장이 동시에 수행되는 패턴은 허용하지 않습니다.

### 위반 탐지 기준

- 함수가 호출하는 외부 모듈이 3개 이상이며 도메인이 다른 경우
- 함수의 반환값과 조작하는 상태 변수 간의 연관성이 낮은 경우
- 함수 내에서 `parse`, `validate`, `save`, `send` 등의 동작이 혼재하는 경우

### 피드백 예시

```
⚠️ [SRP 위반 가능성] 이 함수는 데이터 유효성 검사와 DB 저장 로직을 동시에 수행하고 있습니다.
각각의 책임을 `validateUserInput()`, `saveUserToDatabase()`와 같이 별도 메서드로 분리하는 것을 권장합니다.
```

## 2. 시맨틱 네이밍 (Semantic Naming)

### 규칙

- 변수명과 함수명은 **역할과 의도가 명확히 드러나야** 합니다.
- `data`, `info`, `temp`, `obj`, `val`과 같은 모호한 이름은 사용하지 않습니다.
- 이름만으로 해당 값의 **상태(State)**, **도메인(Domain)**, **용도(Purpose)**를 파악할 수 있어야 합니다.
- 불리언 변수는 `is`, `has`, `can`, `should` 등의 접두어를 사용합니다.

### 네이밍 기준표

| 나쁜 예  | 좋은 예                | 이유                         |
| -------- | ---------------------- | ---------------------------- |
| `data`   | `validatedUserProfile` | 검증 완료 상태와 도메인 명시 |
| `info`   | `pendingOrderRequest`  | 대기 상태와 도메인 명시      |
| `flag`   | `isPaymentConfirmed`   | 불리언 의미 명확화           |
| `list`   | `activeSubscriberList` | 상태와 컬렉션 유형 명시      |
| `result` | `parsedApiResponse`    | 처리 과정과 출처 명시        |

### 피드백 예시

```
💡 [네이밍 개선 제안] 변수명 `data`는 역할이 불명확합니다.
현재 컨텍스트상 `sanitizedInputPayload` 또는 `parsedRequestBody`와 같은 이름이 더 적합합니다.
```

## 3. 코드 복잡도 관리 (Cyclomatic Complexity)

### 규칙

- 함수의 순환 복잡도(Cyclomatic Complexity)는 **10 이하**를 유지해야 합니다.
- 중첩 `if` 문은 **최대 2단계**까지만 허용하며, 그 이상은 조기 반환(Early Return) 패턴으로 변환합니다.
- 복잡한 조건 분기는 **전략 패턴(Strategy Pattern)** 또는 **전용 핸들러 메서드**로 추출합니다.
- `switch`문이 5개 이상의 케이스를 가질 경우, 테이블 기반 처리 또는 전략 패턴 적용을 권장합니다.

### 리팩토링 우선순위

1. 중첩 조건문 → 조기 반환 패턴
2. 복잡한 `switch` → 전략 패턴
3. 반복적인 조건 분기 → 다형성(Polymorphism) 활용
4. 복잡한 루프 → Stream/함수형 처리로 변환

### 피드백 예시

```
🔴 [복잡도 경고] 이 함수의 중첩 if 문이 3단계를 초과합니다.
아래와 같이 조기 반환(Early Return) 패턴으로 개선하면 가독성과 유지보수성이 향상됩니다:

// 개선 전
if (user != null) {
    if (user.isActive()) {
        if (user.hasPermission()) { ... }
    }
}

// 개선 후
if (user == null) return;
if (!user.isActive()) return;
if (!user.hasPermission()) return;
...
```

## 4. 에러 핸들링 및 예외 처리 (Error Handling)

### 규칙

- 모든 외부 API 호출, DB 연산, 파일 I/O는 **예외 처리를 필수로 포함**해야 합니다.
- `catch (Exception e)` 와 같은 포괄적인 예외 처리는 지양하고, **구체적인 예외 타입**을 명시합니다.
- 예외 발생 시 의미 있는 **에러 메시지와 컨텍스트 정보**를 함께 기록해야 합니다.
- `NullPointerException`이 발생할 수 있는 모든 객체 참조는 사전 방어 코드를 포함합니다.
- 파이썬의 경우 **가변 인자를 기본값으로 설정하는 패턴**(`def func(items=[])`)은 허용하지 않습니다.

### 에러 처리 계층 구조

```
[방어적 프로그래밍] → Null 체크, 범위 검사, 타입 검증
         ↓
[Try-Catch 구조]   → 에러 타입별 차별화된 복구 전략
         ↓
[로깅 및 모니터링] → 구조화된 로그 + Cloud Observability 연동
         ↓
[사용자 피드백]    → 의미 있는 에러 메시지 반환
```

### 피드백 예시

```
🔴 [잠재적 NullPointerException] `user.getProfile().getName()` 호출 시
`getProfile()`이 null을 반환할 경우 런타임 오류가 발생합니다.
Optional을 활용하거나 null 체크 로직을 추가해 주세요:

// 권장 패턴 (Java)
Optional.ofNullable(user.getProfile())
        .map(Profile::getName)
        .orElse("Unknown");
```

## 5. 성능 및 자원 효율성 (Performance & Resource Efficiency)

### 규칙

- 루프 내부에서 **불필요한 객체 생성**을 금지합니다.
- N+1 쿼리 패턴은 반드시 **배치 처리 또는 Join 쿼리**로 개선합니다.
- 반복적으로 사용되는 고비용 연산에는 **메모이제이션(Memoization)** 또는 캐싱을 적용합니다.
- 데이터베이스 연결, 파일 스트림, 네트워크 소켓은 사용 후 **반드시 명시적으로 해제**해야 합니다.
- O(n²) 이상의 알고리즘이 감지되면 반드시 최적화 방안을 검토합니다.

### 성능 체크리스트

| 대상        | 감지 패턴                 | 권장 개선 방안                            |
| ----------- | ------------------------- | ----------------------------------------- |
| 시간 복잡도 | 중첩 루프에서의 선형 탐색 | HashMap, Set 등 적절한 자료구조 활용      |
| 메모리      | 루프 내 반복 객체 생성    | 객체 풀링(Pooling) 또는 재사용            |
| DB I/O      | 루프 내 단건 쿼리 반복    | 배치 쿼리 또는 IN 절로 통합               |
| CPU         | 중복 재귀 호출            | 메모이제이션 적용                         |
| 자원 누수   | 미해제 DB 연결/스트림     | try-with-resources 또는 finally 블록 사용 |

### 피드백 예시

```
⚠️ [N+1 쿼리 감지] 이 루프 내에서 매 반복마다 DB 쿼리가 실행되고 있습니다.
`userIds`를 한 번에 조회하는 배치 쿼리로 개선하면 성능이 크게 향상됩니다:

// 개선 전: N번 쿼리 실행
for (Long id : userIds) {
    User user = userRepository.findById(id);
}

// 개선 후: 1번 쿼리 실행
List<User> users = userRepository.findAllByIdIn(userIds);
```

## 6. API 응답 표준화 (API Response Standard)

### 규칙

- 모든 API 응답은 **표준 래퍼 객체(Standard Wrapper Object)**로 감싸야 합니다.
- 성공/실패 여부, 데이터, 에러 메시지를 **일관된 구조**로 반환해야 합니다.
- HTTP 상태 코드는 의미에 맞게 **정확하게 사용**해야 합니다 (200, 201, 400, 401, 403, 404, 500).

### 표준 응답 구조

```json
{
  "success": true,
  "data": { ... },
  "error": null,
  "message": "처리가 완료되었습니다.",
  "timestamp": "2026-03-12T10:00:00Z"
}
```

### 피드백 예시

```
🔴 [API 응답 표준 위반] 이 엔드포인트는 데이터를 직접 반환하고 있습니다.
표준 래퍼 객체(`ApiResponse<T>`)로 감싸서 반환해 주세요.
```

## 7. 코드 가독성 및 자기 완결성 (Readability & Self-Documentation)

### 규칙

- 복잡한 비즈니스 로직에는 **인라인 주석**으로 의도(Intent)를 명확히 설명합니다.
- 비직관적인 수식이나 알고리즘에는 참고 자료 또는 설명 주석을 추가합니다.
- 매직 넘버(Magic Number)는 **명명된 상수(Named Constant)**로 대체합니다.
- 주석은 **"무엇을"이 아닌 "왜(Why)"**를 설명해야 합니다.

### 피드백 예시

```
💡 [가독성 개선 제안] `86400`은 매직 넘버입니다.
의미를 명확히 하기 위해 상수로 추출해 주세요:

// 개선 전
if (elapsedTime > 86400) { ... }

// 개선 후
private static final int SECONDS_PER_DAY = 86400;
if (elapsedTime > SECONDS_PER_DAY) { ... }
```

## 8. 보안 (Security)

### 규칙

- API 키, 비밀번호, 토큰 등의 **민감 정보를 코드에 하드코딩하는 것을 절대 금지**합니다.
- 환경 변수 또는 시크릿 관리 서비스(Secret Manager)를 사용합니다.
- 사용자 입력값은 반드시 **유효성 검사 후 사용**해야 합니다 (SQL Injection, XSS 방지).
- 암호화 알고리즘은 현재 보안 표준에 부합하는 강도를 사용합니다 (MD5, SHA-1 사용 금지).
- 디버그 모드 설정이나 테스트용 인증 우회 코드가 프로덕션 코드에 포함되지 않도록 합니다.

### 피드백 예시

```
🚨 [보안 경고] 코드에 API 키가 하드코딩되어 있습니다.
즉시 환경 변수 또는 Secret Manager로 이동하고, Git 히스토리에서도 제거해 주세요:

// 위반
String apiKey = "sk-1234abcd...";

// 권장
String apiKey = System.getenv("API_KEY");
```

## 9. 테스트 가능성 (Testability)

### 규칙

- 모든 비즈니스 로직 함수는 **단위 테스트(Unit Test)가 가능한 구조**로 작성되어야 합니다.
- 외부 의존성(DB, API, 파일 시스템)은 **인터페이스를 통해 추상화**하여 Mock 처리가 가능해야 합니다.
- 테스트 코드는 **Given-When-Then** 패턴을 따릅니다.
- 예외 상황을 강제로 발생시키는 **네거티브 테스트 케이스**를 반드시 포함합니다.

### 피드백 예시

```
⚠️ [테스트 가능성 저하] 이 함수는 내부에서 DB 연결을 직접 생성하고 있어 단위 테스트가 어렵습니다.
Repository 인터페이스를 주입받는 구조로 변경하여 테스트 시 Mock 객체로 대체할 수 있도록 해주세요.
```

## 10. 피드백 심각도 등급 (Feedback Severity)

모든 코드 리뷰 코멘트는 다음 심각도 등급을 사용합니다:

| 등급     | 아이콘 | 의미                          | 처리 방식                  |
| -------- | ------ | ----------------------------- | -------------------------- |
| Critical | 🚨     | 보안 취약점, 데이터 손실 위험 | 머지 차단, 즉시 수정 필수  |
| High     | 🔴     | SRP 위반, 잠재적 런타임 오류  | 머지 전 수정 강력 권고     |
| Medium   | ⚠️     | 성능 이슈, 복잡도 초과        | 현재 스프린트 내 수정 권고 |
| Low      | 💡     | 네이밍 개선, 가독성 향상      | 다음 리팩토링 시 반영      |
| Info     | ℹ️     | 참고 사항, 대안 제시          | 선택적 적용                |
