# PR 자동 리뷰 스타일 가이드 (Style Guide for Automated PR Review)

## 1. 리뷰 언어 및 형식 (Review Language and Format)

## 1.1. 피드백 언어 (Feedback Language)
- **모든 풀 리퀘스트 요약(Summary) 및 코드 리뷰 코멘트는 반드시 한국어(Korean)로 작성되어야 합니다.**
- 기술 용어는 영어로 유지할 수 있으나, 설명과 피드백 본문은 한국어 문장으로 구성되어야 합니다.

## 1.2. 피드백 태그 사용 (Feedback Tagging)
- 모든 리뷰 피드백은 중요도에 따라 다음 태그 중 하나를 사용하여 시작해야 합니다.
    - **[Critical]**: 반드시 수정해야 하는 버그, 보안 문제, 로직 오류.
    - **[High]**: 성능 저하, 코딩 컨벤션의 심각한 위반, 유지보수성을 크게 해치는 부분.
    - **[Medium]**: 개선하면 좋은 부분, 가독성 향상 제안, 불필요한 코드 제거 제안.
    - **[Low]**: 사소한 스타일 문제나 제안 사항.

---

## 2. 풀 리퀘스트 요약 필수 사항 (Pull Request Summary Requirements)

Gemini Code Assist는 자동 리뷰를 시작하기 전에, **반드시 변경 사항에 대한 포괄적인 요약**을 제공해야 합니다. 이 요약은 다음 항목을 **반드시 포함**해야 합니다.

## 2.1. 기능 요약 (Feature Implemented)
- **변경 목적:** 이 PR이 해결하려는 문제나 구현하려는 **주요 기능**이 무엇인지 명확하게 한 문장으로 요약해야 합니다.
- **주요 변경 파일:** 가장 핵심적인 로직 변경이 일어난 **파일 목록 (최대 5개)**을 명시하고, 각 파일에서 어떤 역할을 수행했는지 간략히 설명해야 합니다.
- **영향 범위:** 이번 변경이 **시스템의 다른 부분에 미치는 잠재적인 영향**을 분석하고 언급해야 합니다. (예: "이 변경은 인증 모듈에 영향을 미칠 수 있습니다.")

## 2.2. 추가 컨텍스트 (Additional Context)
- **테스트 커버리지:** 변경된 코드에 대해 **새로 추가되거나 수정된 테스트**가 있는지 여부를 확인하고 언급해야 합니다. 테스트 커버리지가 불충분하다고 판단될 경우 **[Medium] 경고**를 남겨야 합니다.
- **기술적 특이 사항:** 특정한 라이브러리 사용, 복잡한 알고리즘 적용, 의도적인 트레이드오프(Trade-off) 등 **리뷰어가 주목해야 할 기술적 특이 사항**을 별도로 나열해야 합니다.

## 3. 프로젝트 퀄리티 향상을 위한 규칙 (Code Quality Rules)

## 3.1. 유지보수성 및 리팩토링
- **함수 및 메서드 책임:** 함수나 메서드가 **단 하나의 책임**만 가지도록 리뷰해야 합니다. 하나의 함수가 너무 많은 일을 한다고 판단되면 리팩토링을 권장해야 합니다.
- **중복 코드:** 3줄 이상의 코드가 복사-붙여넣기 된 중복 패턴이 발견되면 **반드시 [High] 경고와 함께 추출할 함수/클래스 제안**을 제공해야 합니다.
- **매직 넘버:** 코드 내에 명확한 설명 없이 사용된 리터럴 숫자는 **상수(Constant)로 추출**하도록 **권고 [Medium]**해야 합니다.

## 3.2. 성능 및 자원 관리
- **불필요한 루프/API 호출:** 반복문 내에서 비효율적인 연산이나 불필요하게 API를 호출하는 코드가 있는지 검토해야 합니다.
- **자원 해제:** 파일 스트림, DB 연결 등 자원을 사용하는 경우, **반드시 자원 해제 로직(`finally` 또는 `close()` 호출)이 명확히 구현**되어 있는지 확인하고 누락 시 **[Critical] 경고**를 남겨야 합니다.

## 3.3. 네이밍 컨벤션 (추가 예시)
- **자바스크립트/타입스크립트:** 변수와 함수 이름은 반드시 `camelCase`를 사용해야 합니다.
- **클래스:** 클래스 이름은 반드시 `PascalCase`를 사용해야 합니다.
- **상수:** 전역 상수는 반드시 `SCREAMING_SNAKE_CASE`를 사용해야 합니다.
- **커밋 메시지 규칙:** PR에 포함된 커밋 메시지는 **Conventional Commits** 규칙을 따르지 않을 경우 **[Low] 경고**를 남겨야 합니다.
